## üìä **Architecture Overview**

### Chosen Architecture: Enhanced SmartFileHandler-Centric

```
Client ‚Üí Enhanced SmartFileHandler ‚Üí UnifiedFileManager ‚Üí Supabase Storage ‚Üí Provider Upload
                                          ‚Üì                           ‚Üì
                                    Embeddings Service (on-demand)  File Metadata
```

### Key Design Decisions

**1. SmartFileHandler as Unified Entry Point**
- ‚úÖ Already has sophisticated decision logic
- ‚úÖ Already has path normalization
- ‚úÖ Already uploads to Kimi
- ‚ûï Add Supabase integration
- ‚ûï Add multi-provider support

**2. Supabase-First Upload Flow**
- ‚úÖ Upload to Supabase for persistence (reliability)
- ‚úÖ Upload to provider for AI processing (functionality)
- ‚úÖ Track both locations in metadata (audit trail)
- ‚úÖ Enables retry without re-upload (idempotency)

**3. On-Demand Embeddings**
- ‚úÖ Generate when similarity search needed
- ‚úÖ Cache in Supabase for reuse
- ‚úÖ Reduces upload time and storage costs
- ‚úÖ Improves performance

**4. Purpose Detection + Fallback**
- ‚úÖ Auto-detect based on content type
- ‚úÖ Try file-extract first for text documents
- ‚úÖ Fallback to assistants if extraction fails
- ‚úÖ Prevents upload failures
